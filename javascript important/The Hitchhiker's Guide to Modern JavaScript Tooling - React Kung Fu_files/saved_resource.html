                                                                Ian Martin Drapeau 3 years ago

                                                                Livereloading and auto-recompile is a fundamentally different experience to write-compile-reload-test. I think that buys a lot of performance. So yes, at least to some extent, you 'lose' that without (or rather, you fail to gain it).

                                                                What transpiling gives you is a bigger language. You have to do less dependency on the common denominator. That, in my experience, can be both a productivity win, or a productivity loss. It depends what you're doing. I find it a maintainability loss, generally, though often only a slight one. The only transpilation that's been a net plus for us in several projects is Less / autoprefixer (i.e. CSS transpilation and adding vendor-prefixes to marginally supported features). Mostly because changing a set of colors (with tints), or a border style throughout a reasonable size CSS set is a PITA, and having coders remember what CSS needs prefixes for what browsers is a drain.

                                                                On React/Flux, I suspect a lot of the answer has to do with papering over Javascript with APIs. Similarly with module systems or ES6 classes. All of which are writable in a few lines of code, but that code means you have to know what Javascript is doing, rather than copying and pasting examples. So, for us, I've not found it worthwhile. Perhaps that will change in future. And it might change depending on who is writing the code.
