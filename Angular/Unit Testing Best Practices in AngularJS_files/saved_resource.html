                                                                      Steve Clements Eddie Monge 4 years ago

                                                                      Just to build on Andy's answer and in the hope that a different explanation of the same thing provides more clarity...:

                                                                      In order to load your module into the tests you must load them with 'angular.mock.module('yourApp');' This is also exposed as a global so, 'module('yourApp');' is exactly the same as 'angular.mock.module('yourApp');'. You would never use angular.module('yourApp');. As Andy says, karma.conf.js contains the reference to the file that contains the module (which in most cases is the actual module your are trying to test).

                                                                      If, in your modules, there are calls to other services or functions that you don't want to test you would create a 'mock' of the service or function. This allows the test system to bypass the call in your module and replace it with the mock version you have defined.

                                                                          Andy Shora Mod Eddie Monge 4 years ago

                                                                          Sorry, I should probably have explained this further, but it's perhaps worthy of a separate article itself.

                                                                          angular.mock.module('App') allows you to load a custom mocked version of the App module. You may require dumbed down behaviour from this module, or you may just want to load the real thing, in which case this call can just simply point to your real module.

                                                                          If you check out the answer on this post, you can see then you can store custom mock versions of modules in a separate folder, which are loaded in the config of your integration test tool (e.g. Karma.conf.js).
                                                                          http://stackoverflow.com/qu...

                                                                          In that example, the mocked module simply references the real module. Even if this is the case, I'd say it's good practice to create mock versions of modules like this in case you want to quickly mock some behaviour that's interfering with your tests.

                                                                          For example, if your 'App' module loads a service called TwitterService which sends http requests to get your last 10 tweets form the Twitter API, you may decide to create a mocked version of the App module which loads a mock version of the TwitterService instead, which simply returns some static JSON.

                                                                          The tests that you're writing which contain this call to angular.mock.module('App') may not be aware of the TwitterService, as it's injected into the App module, but the http request that the TwitterService will send may interfere with your tests. We should therefore configure the App module to suppress this external request for tweets, as this functionality is not important for these particular tests (and should have its own tests elsewhere anyway).

                                                                          TL;DR: It's a good way of configuring modules to run in a test environment, so you can nullify underlying behaviour which may interfere with your tests.
